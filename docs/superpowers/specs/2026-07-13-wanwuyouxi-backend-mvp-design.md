# 万物有戏：后端 MVP 设计

**日期：** 2026-07-13  
**状态：** 已完成产品设计确认，待书面规格终审  
**目标周期：** 2–3 周完成可演示版本，后续持续迭代

## 1. 背景与目标

“万物有戏”将用户上传的一张现实空间照片转化为约三分钟的轻悬疑密室解谜游戏。前端模拟版本已经跑通上传、扫描、查看三条线索、两次答题和结局揭晓流程。

后端 MVP 的目标是把模拟案件替换成真实 AI 生成流程，并满足以下要求：

- 中国大陆网络环境可稳定访问；
- 优先保证秋招演示时的首次生成质量；
- 支持真实照片上传、异步生成、刷新恢复和服务端判题；
- 隔离模型供应商，后续更换模型不影响前端协议；
- 对模型输出执行结构、几何、内容与语义校验；
- 用户照片私有存储、支持立即删除、默认 24 小时内清理；
- 记录生成成功率、热点可用率、延迟和单局模型成本。

MVP 不追求轮廓级物体分割、多人游戏、账号系统、案件分享市场、复杂微服务或模型微调。

## 2. 已确认的技术选择

采用模块化 MVP，而不是一次性接口演示或完整微服务架构。

| 能力 | MVP 选择 | 说明 |
| --- | --- | --- |
| Web 与 API | Next.js App Router + TypeScript | 与现有前端保持单仓库，服务端接口不暴露密钥与答案 |
| 后台生成 | 同仓库独立 Worker 进程 | 数据库任务表驱动，避免请求连接长期占用 |
| 数据库 | PostgreSQL | 保存匿名会话、任务、案件和游玩进度 |
| 开发图片存储 | 本地私有目录适配器 | 便于本地开发，不依赖云资源 |
| 生产图片存储 | 阿里云 OSS 私有 Bucket | 中国大陆访问稳定，使用短期签名 URL |
| 主视觉模型 | 阿里云百炼 `qwen3-vl-plus` 中国内地版 | 非思考模式，负责预检、物体定位和案件草案 |
| 语义校验模型 | DeepSeek 文本模型 | 不接触照片，只校验谜题逻辑并执行定向修复 |
| 数据校验 | Zod + 确定性规则 | 所有模型输出必须经过服务端验证 |

开发和生产通过统一的 `ImageStorage`、`VisionCaseProvider`、`CaseJudgeProvider` 接口切换实现，不在业务逻辑中直接调用某个厂商 SDK。

## 3. 系统边界

后端在一个代码仓库中拆成三个可独立理解的运行单元：

1. **Web/API：** 接收前端请求、建立匿名会话、处理上传、创建任务、查询状态、返回玩家案件和判题；
2. **Generation Worker：** 领取待处理任务、调用模型、执行校验与修复、发布案件；
3. **Cleanup Worker：** 删除过期照片、重试删除失败的对象并记录异常。

核心模块边界：

- `ImageStorage`：保存、读取签名地址和删除照片；
- `GenerationJobRepository`：保证任务幂等和状态单向变化；
- `VisionCaseProvider`：将照片转为固定 Schema 的候选案件；
- `CaseValidator`：执行无需模型的确定性检查；
- `CaseJudgeProvider`：检查答案唯一性、线索一致性和故事自洽性；
- `CasePublisher`：仅发布通过全部检查的案件，并生成不含答案的玩家视图；
- `GameSessionService`：记录线索和答题状态，执行服务端权威判题。

每个模块只依赖显式接口。模型、存储或数据库实现变化时，前端 API 和游戏状态机保持不变。

## 4. 端到端数据流

1. 前端将照片压缩为最长边不超过 1600 px、文件不超过 5 MB 的 JPEG；
2. API 校验文件签名、尺寸和内容类型，将照片写入私有存储；
3. API 使用匿名会话、图片摘要和幂等键创建生成任务，立即返回 `jobId`；
4. 前端进入扫描页并轮询任务状态，不保持长连接；
5. Worker 领取任务并生成有效期不超过 5 分钟的图片访问地址；
6. `qwen3-vl-plus` 以非思考模式返回照片预检、候选物体、归一化热点坐标和案件草案 JSON；
7. 服务端解析 JSON，并依次执行 Schema、长度、枚举、坐标、重叠、物体绑定、唯一答案和内容安全检查；
8. DeepSeek 只接收脱敏后的案件 JSON 和可见物体名称，判断线索是否共同支持唯一答案；
9. 若存在可修复问题，最多执行一次定向修复，并重新执行完整校验；
10. 全部通过后写入不可变案件，任务进入 `SUCCEEDED`；否则进入可解释的失败状态；
11. 前端获取不含正确答案和真相的玩家案件，开始确定性游玩；
12. 用户提交答案时，服务端检查所属会话、剩余次数并权威判题；
13. 揭晓后才返回真相文本。游玩阶段不再调用模型。

## 5. 模型职责与调用预算

### 5.1 千问视觉生成

`qwen3-vl-plus` 负责：

- 判断图片是否为可用的现实室内或近距离空间；
- 判断亮度、清晰度、物品数量和输入安全性；
- 选择三个明显、可区分、易点击且无需读取小字的物品；
- 返回三个物品的归一化中心点、点击半径和定位置信度；
- 根据三个真实物品生成固定题型的轻悬疑案件；
- 严格返回约定 JSON，不输出 Markdown 或额外说明。

默认最多调用一次。仅当首次响应是临时网络错误、限流或可修复的格式错误时允许重试一次，因此每个任务最多两次千问调用。

### 5.2 DeepSeek 语义校验

DeepSeek 负责：

- 判断三条线索是否共同支持唯一答案；
- 判断错误选项是否不会同样成立；
- 判断故事、线索和真相是否一致；
- 针对明确问题修复指定字段，不重新自由创作整个案件。

DeepSeek 不接收照片、签名 URL、对象 Key、匿名会话标识或用户输入文本。每个任务最多一次校验和一次修复调用。

### 5.3 DeepSeek 降级

DeepSeek 超时或不可用时，不把未经检查的复杂案件直接发布。仅当案件通过全部确定性检查，并且视觉模型同时返回高逻辑置信度时，允许以 `judge_degraded=true` 发布；否则任务进入可重试失败。所有降级必须记录指标，便于后续评估是否收紧策略。

## 6. 输出不变量与热点降级

案件发布前必须同时满足：

- 三个线索物品名称唯一，且均来自照片中识别到的物品；
- 三个坐标均在 `[0, 1]`，点击半径处于允许范围；
- 热点不存在严重重叠；
- 只有一个正确答案；
- 线索不依赖照片中的小字、真实人物身份或照片外关键证据；
- 内容不包含违法指导、色情、血腥细节、自残、仇恨或真实人物犯罪指控；
- 标题、案情、线索、选项、提示和真相均符合长度限制；
- 模型输出不存在 Schema 外的可执行内容。

若案件语义合格，但任一热点定位置信度不足或几何校验失败，案件不重新生成，改为 `CARD_FALLBACK`。前端保留照片背景，并在照片下展示三个物品卡片。该降级仍计为生成成功，但单独计入热点可用率指标。

## 7. 状态、幂等与恢复

生成任务状态只允许按以下方向变化：

`PENDING -> PROCESSING -> VALIDATING -> SUCCEEDED | RETRYABLE_FAILED | REJECTED | FAILED`

- `RETRYABLE_FAILED` 可由用户主动重试，并创建新尝试记录；
- `REJECTED` 表示图片不适合或不安全，不复用原任务；
- `FAILED` 表示超过重试预算或出现不可恢复内部错误；
- 已成功案件不可修改，修复只能发生在发布之前；
- Worker 通过数据库租约领取任务，租约过期后其他 Worker 可安全接管；
- 相同匿名会话、图片摘要和幂等键只创建一个有效任务；
- 前端等待 30 秒后可离开扫描页，后台任务继续；刷新或返回时通过 `jobId` 恢复；
- 重复点击生成按钮不会重复调用模型或扣费。

游戏进度保存已查看线索、答题次数、最后提交选项和完成状态。每局最多提交两次答案，所有状态变更由服务端验证。

## 8. 失败与用户提示

### 8.1 图片不合格

画面太暗、模糊、不是现实空间、可用物品少于三个时，返回具体、可行动的拍摄建议。敏感内容只显示统一拒绝提示，不回显识别细节。

### 8.2 网络和模型异常

- 临时超时、连接失败或限流：后台自动重试一次；
- 非法 JSON 或字段小范围错误：允许一次定向修复；
- 连续失败：保留浏览器本地照片预览，用户可重新生成，无需重新选择；
- 用户侧只显示易懂提示和 `traceId`，不显示供应商、Prompt、数据库错误或堆栈。

### 8.3 内容不合格

答案不唯一、线索矛盾、依赖照片外信息或包含不安全内容时不发布。一次修复后仍失败，提示用户更换拍摄角度或场景。

### 8.4 调用上限

单任务最多两次千问调用和两次 DeepSeek 调用。达到上限后停止自动尝试，防止费用和延迟失控。

## 9. 数据与隐私

- 用户无需注册；服务端使用 HttpOnly、Secure、SameSite=Lax Cookie 标识匿名会话；
- 图片使用私有对象，禁止公共读，不接受客户端提供的任意远程图片 URL；
- 模型读取地址有效期不超过 5 分钟；
- 原图默认在上传后 24 小时内删除，用户主动删除时立即进入幂等删除流程；
- 案件可在照片删除后保留，但必须与图片访问能力解绑；
- 不使用用户照片训练模型或建立内部数据集；
- 日志不得记录 Base64、图片地址、对象 Key、模型识别文本、完整剧情或 API Key；
- 数据库记录任务状态、错误码、模型名称、Token、耗时和估算费用；
- 正确答案只保存在服务端，揭晓前的任何 API 响应都不得包含答案索引或真相；
- `.env.local` 保存本地密钥并被 Git 忽略；仓库只提交无密钥的 `.env.example`；
- 生产密钥由部署平台加密环境变量管理，绝不发送到浏览器。

## 10. 测试策略

### 10.1 自动化测试

- **单元测试：** Schema、文本长度、坐标、热点重叠、唯一答案、状态迁移、幂等和答题次数；
- **接口测试：** 会话、上传、任务创建与查询、失败重试、玩家案件、判题、揭晓和删除照片；
- **模型契约测试：** 固定千问与 DeepSeek 响应，覆盖合法输出、非法 JSON、缺字段、超时、限流和修复失败；
- **Worker 测试：** 任务租约、崩溃接管、重复执行和终态不可回退；
- **端到端测试：** 从上传照片到完成案件，并覆盖刷新恢复和卡片降级。

自动化测试默认使用 Fake Provider，不消耗真实模型额度。真实模型测试必须显式开启。

### 10.2 真实图片评测

先使用 10 张授权照片调试，稳定后扩充到至少 50 张，覆盖客厅、卧室、书桌、办公室、咖啡店、杂乱场景、暗光和物品过少场景。每张照片建立人工标注：是否可生成、三个物体是否真实可见、热点是否可点击、答案是否唯一、内容是否安全。

记录：

- 合格照片的案件生成成功率；
- 热点位置可用率和卡片降级率；
- P50/P95 生成时长；
- 千问与 DeepSeek 的调用次数、Token 和单局平均成本；
- 修复率、重试率和各类失败原因。

## 11. MVP 验收标准

- 至少 80% 的合格评测照片可生成可玩的案件；
- 成功任务通常在 30 秒内进入游戏，超时后用户可以离开并恢复；
- 非法 Schema、严重热点重叠、答案不唯一或不安全案件不会发布；
- 定位不可靠但语义合格的案件自动进入卡片模式；
- 刷新页面可恢复生成任务和游戏进度；
- 揭晓前客户端无法获取正确答案；
- 每局最多提交两次答案；
- 模型或网络失败后存在明确的重试或换图操作；
- 照片支持立即删除，并在 24 小时内自动过期；
- 日志、埋点和 API 响应不泄露照片、密钥、答案或内部错误；
- 中国大陆手机网络能够完成一次上传、生成、游玩和删除；
- 形成成功率、热点可用率、生成时长和单局成本四项可用于简历展示的数据。

## 12. 实施顺序

1. 建立共享 Schema、数据库模型、匿名会话和 Fake Provider；
2. 实现本地图片存储、上传、任务状态和独立 Worker；
3. 将现有前端模拟流程切换到真实 API，但仍使用固定案件；
4. 接入 `qwen3-vl-plus`、结构化输出与确定性校验；
5. 接入 DeepSeek 校验、一次修复和降级策略；
6. 完成服务端判题、刷新恢复、删除和清理任务；
7. 建立 10 张再到 50 张的真实图片评测集；
8. 接入阿里云 OSS 和 PostgreSQL，完成中国大陆真机验证；
9. 补齐成本、延迟、失败原因和隐私验收报告。

## 13. 明确排除项

本轮不实现：

- 用户账号、手机号、微信登录；
- 多张照片生成一个房间；
- 多人协作、对战和公开案件链接；
- 视频输入、AR、物体轮廓分割；
- 自定义题材、难度和时长；
- 案件编辑器、内容社区和排行榜；
- 模型微调、向量数据库和知识库；
- Kafka、Kubernetes 或独立微服务拆分；
- 用户照片长期保存或用于训练。
