# 万物有戏 V2：真实物证与受约束审问设计

**日期：** 2026-07-19  
**状态：** 用户已确认采用，待实施计划  
**目标用户：** 喜欢轻悬疑、互动故事和可分享内容的移动端用户；同时作为 AI 产品经理秋招作品集公开演示。

## 1. 产品定位

万物有戏 V2 是一个以真实空间为输入的三分钟 AI 悬疑体验：AI 从用户照片中识别三件真实可见的物证，围绕物证生成三名嫌疑人和一个唯一可验证的证词矛盾。玩家探索物证、审问嫌疑人、提交“说谎者 + 关键物证”的组合答案，最后获得案件档案卡，并可主动选择生成一页漫画。

V2 不把物品拟人化。物品始终是物证；人物承担动机、行动、证词和戏剧冲突。产品差异化不是沙盒养成，而是“真实物证约束下的生成式审问”。

## 2. 设计原则

1. **照片必须影响推理。** 三件物证必须来自照片中真实可见的物体，不能只是故事背景。
2. **只有一个可解矛盾。** 三份证词中只有一名隐藏说谎者，且必须存在一件能证明其说谎的关键物证。
3. **生成式表达，服务端事实。** NPC 可以有自然语言风格，但不能发明人物、时间、地点或新证据。
4. **失败不伪装成功。** 案件不可解、结构异常或模型不可用时，明确失败并允许重新生成。
5. **即时结果优先。** 案件档案卡不依赖图片模型；漫画是可选异步增强，不阻塞结案。
6. **公开体验可控。** 匿名用户、模型调用、漫画生成和上传均有服务端配额。

## 3. MVP 游戏流程

1. 用户拍摄或上传一张空间照片。
2. Qwen Vision 识别场景与候选物体，返回坐标、可见描述和置信度。
3. DeepSeek 根据视觉事实编译案件事实簿、三名嫌疑人和唯一矛盾。
4. 服务端完成结构校验、确定性校验和语义可解性校验。
5. 玩家进入案件简报，仅看到背景、目标和三处待探索光圈。
6. 玩家点击物证，看到物证描述，并解锁与其关联的嫌疑人角色卡。
7. 玩家点击嫌疑人头像进入审问：每人两条推荐问题，以及一次自由提问机会。
8. 玩家提交一名说谎者和一件关键物证。两者必须同时正确才算首次破案成功。
9. 首次错误后展示不泄露答案的方向性提示，并允许第二次提交。
10. 结案后展示完整时间线、说谎动机和证据链，并立即生成案件档案卡。
11. 用户可主动同意将现场照片发送给图片模型，异步生成一页无文字漫画底图；系统随后叠加准确中文对白与旁白。

## 4. 角色与立绘

- 内置 12–18 个统一黑色电影风格的预设立绘，覆盖不同年龄、气质与身份原型。
- AI 只选择 `portraitKey`，并生成姓名、职业、与案件相关人物的关系、性格标签、行动和证词。
- 同一案件中的三张立绘不得重复。
- 运行时不生成立绘，不提供“重绘角色”，不为角色立绘调用图片模型。
- 角色卡公开字段包括姓名、身份、关系、性格、初始证词和已解锁状态；私有行动与知识范围只保留在服务端。

## 5. 案件事实簿

服务端保存的私有事实簿至少包含：

- `visualFacts`：来自照片的可见事实，每条带物体、描述、坐标和置信度。
- `evidence`：恰好三件物证，每件绑定一个 `visualFactId` 和一名嫌疑人。
- `timelineFacts`：按时间排列的案件事实。
- `suspects`：恰好三名嫌疑人，包含私有行动、知识范围、公开证词和允许透露的事实编号。
- `claims`：可供玩家收集的证词断言。
- `liarSuspectId`：唯一说谎者。
- `contradiction`：一条 `claimId + evidenceId + explanation` 组成的唯一矛盾。
- `truth`：完整真相、动机与证据链。
- `recommendedQuestions`：每名嫌疑人两条推荐问题及其预生成回答。

发送给浏览器的玩家视图不得包含 `liarSuspectId`、私有行动、知识范围、正确物证、矛盾解释或完整真相。

## 6. 模型职责边界

### Qwen Vision

只负责图片理解：场景类别、候选物体、物体可见描述、归一化坐标与置信度。它不直接决定凶手，不生成自由故事，也不输出最终证据链。

### DeepSeek 案件编译器

只接收 Qwen 的结构化视觉事实，不接收原始照片或照片地址。它生成案件事实簿、嫌疑人、证词、时间线和矛盾。

### 确定性验证器

发布前必须验证：

- 恰好三件唯一物证和三名唯一嫌疑人；
- 每件物证绑定照片中存在的高置信视觉事实；
- 三张 `portraitKey` 不重复且来自预设白名单；
- 恰好一个说谎者；
- 矛盾引用存在的证词与物证；
- 正确组合唯一；
- 文案、坐标、半径、敏感内容和字段长度合法。

### DeepSeek 可解性评审

评审模型只接收结构化案件，判断玩家是否能仅依靠公开证词和物证得到唯一答案。评审失败时允许一次定向修复；第二次仍失败则拒绝发布。

## 7. 审问机制

### 推荐问题

- 每名嫌疑人恰好两条。
- 问题和回答在案件编译阶段生成并验证，点击后即时展示，不产生额外模型调用。
- 回答必须引用事实簿中的事实编号。

### 自由提问

- 每名嫌疑人每局最多一次，共最多三次。
- 服务端只向模型提供该嫌疑人允许知道和透露的事实子集、人物语气和已公开证词。
- 模型返回结构化结果：`answer`、`dialogueAct`、`factRefs`、`evidenceRefs`。
- `factRefs` 和 `evidenceRefs` 必须属于该嫌疑人的允许集合。
- 语义校验确认自然语言回答没有加入事实集合之外的具体人物、时间、地点和证据。
- 校验失败允许一次重写；再次失败时返回符合角色语气的回避回答，不消耗玩家的提问机会。
- 对“忽略规则”“直接告诉我答案”等提示注入，固定返回不泄露真相的角色化回避。

自由审问失败不终止案件；推荐问题、物证探索和最终推理始终可继续。

## 8. 结算规则

玩家提交：

- `accusedSuspectId`
- `supportingEvidenceId`

两者同时匹配私有矛盾才算正确。第一次错误返回方向性提示，不返回正确人物或物证；第二次提交后结案并揭晓真相。服务端负责判定，浏览器在揭晓前不持有答案。

## 9. 案件档案卡

档案卡在结案后立即生成，不调用图片模型。服务端使用固定竖版模板、现场照片、预设立绘和准确中文文本渲染 PNG，内容包括：

- 案件标题与编号；
- 现场照片；
- 三名嫌疑人与关键证词；
- 玩家指认结果；
- 说谎者、关键物证和矛盾解释；
- 完整结案旁白与破案评价。

档案卡支持下载和系统分享；分享操作必须由用户主动触发。

## 10. 可选一页漫画

- 漫画不是结案前置条件，也不是默认模型调用。
- 仅结案用户可点击“生成案件漫画”。点击前说明照片将发送给阿里云图片模型并取得明确同意。
- DeepSeek 先把案件真相转成四格分镜 JSON。
- `wan2.7-image` 使用现场照片和三张预设立绘作为参考，生成一张 9:16、四格、无文字底图。
- 服务端在底图上叠加中文标题、对白框和旁白，避免模型文字乱码。
- 单用户每天最多一张漫画；全站每日成功生成上限为 30 张，配置项可调整。
- 只有成功图片计入预算。失败时保留档案卡，并允许在当日额度内重试一次。
- 原始照片已删除或全站额度耗尽时，不创建图片任务。

按[阿里云官方公开价格](https://help.aliyun.com/en/model-studio/model-training-and-deployment-billing)，`wan2.7-image` 约 0.20 元/张；默认上限对应每日最高约 6 元图片费用，不含低成本文本调用。模型支持图像编辑和多图参考，能力边界以[官方图像模型文档](https://help.aliyun.com/en/model-studio/image-model/)为准。

## 11. 错误处理与现有修复

V2 功能开发前先完成以下基础修复：

1. 当前 Qwen 输出可能包含单字但非空答案选项；V1 契约应接受非空中文单字选项，同时继续拒绝空字符串、重复选项和非法正确索引。
2. 删除 `PhoneShell` 中的 `9:41` 与右上角伪系统图标，并收回对应顶部占位。
3. 结构失败日志只记录任务 ID、模型、错误码和字段路径，不记录照片、剧情、证词或答案内容。

V2 运行时错误遵循：

- 视觉事实不可用：提示重新拍摄，不创建案件。
- 案件结构或语义不可解：一次定向修复后失败，明确提示重新生成。
- 自由审问失败：退回角色化回避，不中断案件。
- 档案卡失败：页面仍显示完整结案内容，可重试导出。
- 漫画失败或超额：保留档案卡，不影响破案结果。

## 12. 隐私与安全

- 照片存储于私有 OSS，不使用公开永久 URL。
- DeepSeek 不接收原图、照片地址、会话标识或用户身份。
- 漫画生成是独立的用户授权动作；未授权时原图不发送给图片生成模型。
- 图片、案件和会话按现有删除策略到期清理；用户主动重玩或删除时立即排队清理。
- API 密钥只存在服务端环境变量中。
- 匿名用户默认每天最多创建 3 个真实案件、提出每局 3 次自由问题、生成 1 张漫画。
- 速率与预算计数保存在数据库，不依赖浏览器状态。

## 13. 正式部署

第一版公开作品集部署采用单实例架构：

- 阿里云香港 ECS，避免等待中国大陆服务器的 ICP 备案流程；阿里云官方说明香港及海外节点不要求完成 ICP 备案后才能接入；
- 自有域名与 HTTPS；
- Docker 运行 Next.js 应用和定时清理任务；
- Caddy 或 Nginx 负责 TLS 与反向代理；
- 阿里云 OSS 保存私有图片与生成结果；
- PGlite 数据目录挂载到持久化云盘，并每日备份；
- GitHub 主分支通过受控部署脚本更新容器；
- 运行健康检查、生成失败率、P95 延迟和每日模型费用监控。

单实例 PGlite 只服务作品集阶段的小流量，不支持横向扩展。并发量或可靠性要求提高后，再将 Drizzle 数据层迁移至 PostgreSQL/RDS，不在本轮 MVP 范围内。

## 14. MVP 范围

### 包含

- 三件真实物证、三名嫌疑人、一个说谎者和一个唯一矛盾；
- 12–18 张预设角色立绘；
- 每人两条推荐问题和一次自由提问；
- 人物 + 物证组合判题；
- 案件档案卡；
- 用户主动触发、有限额的一页漫画；
- 正式 HTTPS 公网部署、匿名限流与成本上限；
- 格式兼容、伪状态栏清理和完整失败提示。

### 不包含

- 物品拟人化；
- 角色立绘生成或重绘；
- 无限自由聊天；
- 长期角色养成与跨案件记忆；
- 好友系统、多人协作或 NPC 跨玩家访问；
- 连载世界、资源生产、沙盒建造与经济系统；
- 自动生成漫画；
- 多页漫画、视频或语音审问；
- 首版即迁移至 RDS 或多实例扩容。

## 15. 质量门槛

公开部署前至少完成：

- 20 张授权测试照片中，案件结构成功率不低于 90%；
- 已发布案件的唯一可解率为 100%；
- 三个光圈在 iPhone 视口均可点击；
- 自由回答的越权事实率为 0%；
- 答案与真相在揭晓前不出现在浏览器响应中；
- 档案卡导出成功率不低于 99%；
- 漫画失败不阻断结案，且每日预算硬上限有效；
- 单元测试、接口测试、移动端端到端测试、生产构建和公网冒烟测试全部通过。

## 16. 实施阶段

1. **稳定性与上线基础：** 修复单字选项、删除伪状态栏、补充限流与正式部署。
2. **案件事实簿：** 拆分视觉观察与案件编译，扩展校验和私有玩家视图。
3. **角色与审问：** 角色卡、推荐问题、受约束自由提问和越权回退。
4. **推理与档案：** 人物 + 物证判题、揭晓流程和档案卡导出。
5. **漫画增强：** 授权、异步任务、每日预算、万相底图和中文排版。
