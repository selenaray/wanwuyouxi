# 万物有戏 MVP 产品需求文档

> 版本：V1.0  
> 日期：2026-07-11  
> 状态：已确认
> 目标周期：2–3 周

## 1. 产品概述

### 1.1 一句话定义

上传一张现实空间照片，AI 利用照片中真实存在的物品，生成一场 3 分钟可完成的悬疑密室解谜。

### 1.2 核心价值

用户不需要输入提示词或学习创作工具。一次普通拍照即可把熟悉的现实空间转化为可探索、可推理、可完成的悬疑现场。

产品的核心魔法时刻是：用户在游戏中看到的线索，确实来自自己刚刚拍摄的照片。

### 1.3 产品原则

1. AI 负责理解照片与创作内容，程序负责保证游戏可完成。
2. 一局只有一个目标、三个线索和一个唯一答案。
3. 用户首先感受到“这是我的空间”，其次才是“这是 AI 生成的”。
4. 生成失败必须有明确解释和可执行的下一步，不让用户面对空白页。
5. MVP 优先保证稳定、快速和精致，不追求题型数量与内容长度。

## 2. 目标与非目标

### 2.1 MVP 目标

- 跑通“上传照片—生成案件—探索三个线索—完成推理—获得结局卡”的完整闭环。
- 验证用户是否认为线索与自己的照片存在真实、合理的联系。
- 验证用户是否愿意完成一局，并再次拍摄另一个空间。
- 形成适合 AI 产品经理秋招展示的可用产品、用户反馈与数据证据。

### 2.2 非目标

MVP 不包含：

- 实时摄像头识别、AR、3D 空间重建；
- 多张照片或跨房间探索；
- 视频、动画或 AI 配图生成；
- 多人联机、排行榜、积分、成就；
- 开放式自由对话或无限分支剧情；
- 用户编辑、发布或交易关卡；
- 内容社区、关注、评论；
- 付费系统；
- 原生 App；
- 长期保存用户原始照片。

## 3. 目标用户与场景

### 3.1 目标用户

- 喜欢轻度解谜、密室逃脱或悬疑内容的年轻用户；
- 愿意尝试 AI 新产品；
- 使用手机完成体验；
- 没有专业游戏经验，也能在 3 分钟左右完成一局。

### 3.2 推荐拍摄场景

- 桌面、宿舍、卧室、办公室、咖啡馆角落；
- 光线充足；
- 画面中存在至少 5 个可辨识物体；
- 物品之间有一定空间距离，避免完全遮挡。

### 3.3 不推荐场景

- 纯白墙、天空、地面等物体过少的照片；
- 严重过暗、过曝、模糊或抖动照片；
- 以人脸、证件、屏幕聊天记录为主体的照片；
- 截图、表情包、插画或与现实空间无关的图片；
- 包含明显违法、色情、血腥或危险内容的图片。

## 4. MVP 游戏结构

### 4.1 固定题型

MVP 只支持“失踪事件”。玩家根据现场遗留的三个线索，判断失踪者去了哪里或发生了什么。

每局固定包含：

- 1 个案件标题；
- 1 段不超过 80 字的案件背景；
- 3 个来自照片的可探索物品；
- 每个物品对应 1 条不超过 50 字的线索；
- 1 道最终推理题；
- 3 个候选答案；
- 1 个唯一正确答案；
- 1 段不超过 120 字的真相说明。

### 4.2 游戏规则

- 三个线索点在案件开始时均可探索，不规定顺序。
- 玩家必须查看全部三个线索后，才能进入最终推理。
- 每个线索只需点击一次；重复点击可重新查看，不重复计数。
- 玩家选择答案后需二次确认，避免误触。
- 每局最多作答两次。
- 首次答错时不直接揭晓答案，提供一次文字提示。
- 第二次答错或首次答对后，展示真相。
- 游戏不设置失败终止，所有用户最终都能看到结局。

## 5. 核心用户流程

### 5.1 首页

页面内容：

- 产品名与一句话价值主张；
- 一张示例现场及简短玩法演示；
- 主按钮“开始扫描现场”；
- 次级入口“使用示例照片体验”；
- 隐私说明入口。

要求：首次进入的用户应在 5 秒内理解需要拍摄一个包含多个物品的现实空间。

### 5.2 拍摄与上传

支持：

- 调用手机摄像头拍照；
- 从相册选择一张图片；
- 上传前裁剪；
- 删除并重选。

上传前提示：

- 保证光线充足；
- 画面中包含 5 个以上物品；
- 避免拍摄人脸、证件和私人信息。

限制：

- 仅支持 JPEG、PNG、HEIC；
- 客户端统一转换为 JPEG；
- 压缩后最长边不超过 1600 px；
- 压缩后文件不超过 5 MB；
- 一次只处理一张图片。

### 5.3 照片预检

预检依次检查：

1. 文件是否合法、能否解码；
2. 图片尺寸是否足够；
3. 是否严重模糊、过暗或过曝；
4. 是否为现实空间照片；
5. 是否包含足够数量的可辨识物体；
6. 是否包含禁止内容或明显隐私风险。

结果分为：

- `PASS`：进入案件生成；
- `RETRY`：给出具体重拍建议；
- `BLOCK`：因安全原因拒绝处理。

### 5.4 生成等待

- 上传完成后立即进入扫描动画；
- 2 秒内显示第一条状态反馈；
- 状态文案只描述真实阶段，例如“识别现场物品”“正在组织线索”；
- 不展示虚假的具体识别结论；
- 生成超过 15 秒时显示“比平时多花一点时间”；
- 生成超过 30 秒自动结束并进入重试页；
- 同一照片允许一键重试一次，不重复上传。

### 5.5 案件介绍

- 显示案件标题、背景和目标；
- 背景文字不超过三屏行；
- 点击“进入现场”后开始计时；
- 不要求用户阅读玩法教程，以一次手势提示说明“点击照片中的异常区域”。

### 5.6 现场探索

- 原照片作为主要交互画面；
- 三个线索点使用低强度呼吸光圈提示；
- 光圈表示可探索范围，不精确描绘物体轮廓；
- 10 秒无操作后，未发现的线索点增强一次亮度；
- 30 秒无操作后，显示文字提示“试试桌面附近的物品”；
- 点击线索点后，以底部卡片展示物品名和线索；
- 已查看的线索点变为已收集状态；
- 底部固定显示“线索 1/3”；
- 收集全部线索后出现“开始推理”按钮。

### 5.7 最终推理

- 展示问题和三个答案；
- 可展开回看已收集线索；
- 选择后点击“提交推理”；
- 首次答错展示不超过 30 字的提示，并允许再次选择；
- 第二次提交后进入真相页。

### 5.8 真相与结果卡

- 展示是否首次答对、用时和案件真相；
- 程序使用固定模板绘制结果卡，不调用图片生成模型；
- 结果卡不默认包含用户原始照片，只使用模糊化的色彩背景或抽象裁切；
- 提供“保存案件卡”和“再拍一个现场”；
- Web Share API 可用时提供系统分享，不可用时仅保存图片。

## 6. AI 生成与校验

### 6.1 处理流程

1. 安全与质量检查；
2. 场景识别与候选物体提取；
3. 选择三个适合点击、位置分散的物体；
4. 根据固定失踪事件结构生成案件；
5. 程序进行结构校验；
6. AI 进行语义自洽校验；
7. 通过后生成不可变游戏状态；
8. 校验失败时定向修复一次；
9. 再次失败时使用安全模板或提示用户重拍。

### 6.2 物体选择规则

三个线索物体必须：

- 确实存在于照片中；
- 名称能被普通用户理解；
- 大致位置可定位；
- 不是人脸、人体敏感部位、证件或私人文本；
- 交互区域之间不存在大面积重叠；
- 至少分布在照片的两个不同区域；
- 不依赖读取照片中无法可靠识别的小字。

### 6.3 结构化输出要求

AI 必须返回符合固定 Schema 的数据，至少包括：

- 场景类型与质量置信度；
- 三个物体的名称、归一化坐标和点击半径；
- 案件标题、背景、任务；
- 三条线索；
- 最终问题；
- 三个答案和正确答案索引；
- 首次答错提示；
- 真相说明；
- 内容风险标签；
- 生成置信度。

任何字段缺失、超长、类型错误或坐标越界均视为校验失败。

### 6.4 确定性校验

程序至少检查：

- 正确答案索引只能指向一个选项；
- 三个答案文本不重复；
- 三个物体名称不重复；
- 坐标均在图片范围内；
- 点击区域不严重重叠；
- 所有文本满足长度限制；
- 不含未替换变量、Markdown 代码块或系统提示内容；
- 每条线索都绑定一个物体；
- 游戏状态生成后不可被客户端修改正确答案。

### 6.5 语义校验

语义校验模型判断：

- 三条线索是否共同支持正确答案；
- 错误答案是否具有迷惑性但不会同样成立；
- 真相是否能解释全部线索；
- 是否凭空引入照片外的关键证据；
- 是否出现明显逻辑矛盾；
- 内容是否适合轻悬疑定位。

低置信度结果不直接上线，执行一次定向修复。

## 7. 边界情况与降级策略

### 7.1 图片输入

| 边界情况 | 产品处理 |
| --- | --- |
| 文件损坏或格式无法解析 | 阻止上传，提示重新选择图片 |
| 图片尺寸过小 | 提示使用原图或重新拍摄 |
| 文件过大 | 客户端压缩；压缩失败则提示换图 |
| HEIC 转换失败 | 保留当前页，提示拍照或上传 JPEG/PNG |
| 严重模糊、过暗、过曝 | `RETRY`，展示对应拍摄建议 |
| 物体少于 5 个 | `RETRY`，建议拍摄桌面或房间全景 |
| 照片中只有一个大型物体 | `RETRY`，提示扩大取景范围 |
| 截图、插画、表情包 | `RETRY`，说明目前仅支持现实空间照片 |
| 多人合照或人脸为主体 | `RETRY`，提示避开人物重新拍摄 |
| 存在少量路人或非主体人脸 | 自动排除人体区域，不将其作为线索 |
| 包含证件、银行卡、二维码 | `BLOCK` 或要求重拍，不解析敏感文字 |
| 包含聊天记录或私人文本 | `RETRY`，提示遮挡敏感信息 |
| 色情、血腥、违法内容 | `BLOCK`，不生成游戏 |
| 重复上传同一照片 | 允许重新生成，但提示“将创建一个新案件” |

### 7.2 物体定位与点击

| 边界情况 | 产品处理 |
| --- | --- |
| 坐标越界 | 服务端校验失败并修复，不下发客户端 |
| 三个热点重叠 | 重新选择物体；仍失败则使用物品卡片模式 |
| 物体太小 | 扩大点击半径，但不超过图片短边的 12% |
| 点击区域与视觉物体略有偏移 | 使用容错光圈，不要求轮廓级精度 |
| 横竖屏裁切导致坐标错位 | 坐标基于原图归一化，按 `object-fit: contain` 映射 |
| 用户始终找不到线索 | 10 秒增强光圈，30 秒提供区域提示 |
| 模型无法可靠定位 | 降级为照片下方三张物品卡片，仍保留照片背景 |

### 7.3 案件内容

| 边界情况 | 产品处理 |
| --- | --- |
| Schema 不合法 | 自动修复一次；失败则终止本次生成 |
| 正确答案不唯一 | 校验不通过，重新生成答案与线索 |
| 线索与照片无关 | 语义校验不通过，定向重写 |
| 依赖读取模糊小字 | 删除该证据并替换物体 |
| 内容过长 | 服务端截断不可取；要求模型按长度重写 |
| 生成真实人物犯罪指控 | 改写为虚构角色，不使用照片中人物身份 |
| 恐怖程度过高 | 降级为轻悬疑，不描写伤害细节 |
| 与已有案件文本高度重复 | 允许 MVP 阶段存在结构相似，但标题与核心线索需重写 |
| 内容置信度低 | 不展示，尝试修复或返回重拍建议 |

### 7.4 网络与系统

| 边界情况 | 产品处理 |
| --- | --- |
| 上传中断 | 保留本地图片，提供“重新上传” |
| 生成接口超时 | 30 秒终止，允许原图重试一次 |
| 模型服务限流 | 显示“当前体验人数较多”，稍后重试 |
| 重复点击生成按钮 | 使用请求幂等键，只创建一个任务 |
| 用户离开页面再返回 | 任务未过期时恢复当前进度 |
| 游戏中刷新页面 | 通过会话 ID 恢复已生成游戏与已收集线索 |
| 会话已过期 | 返回首页，说明照片已删除，需要重新拍摄 |
| 答案重复提交 | 服务端只接受当前允许次数内的提交 |
| 分享 API 不支持 | 降级为保存结果卡 |
| 结果卡生成失败 | 仍展示文本结局，提供再次生成卡片 |

### 7.5 用户行为

| 边界情况 | 产品处理 |
| --- | --- |
| 未看线索直接推理 | 按钮保持不可用并说明需收集 3 条线索 |
| 快速连续点击多个热点 | 逐条展示，已收集状态不重复计数 |
| 点击照片非热点区域 | 不惩罚；首次可显示轻微扫描反馈 |
| 返回上一页重新选图 | 二次确认后废弃当前案件 |
| 游戏停留时间过长 | 30 分钟后提示会话即将过期 |
| 用户连续答错 | 第二次后揭晓真相，不阻断完成流程 |

## 8. 安全与隐私

### 8.1 隐私原则

- 上传前明确说明照片会被用于生成本次游戏；
- 默认不要求注册；
- 原始照片仅短期保存，默认在上传后 24 小时内自动删除；
- 用户主动结束案件时可触发立即删除；
- 分析结果与匿名行为数据可以保留，但不得包含原始照片或可识别私人文本；
- 不使用用户照片训练模型；
- 结果卡默认不展示清晰原图；
- 日志不得记录图片 Base64、签名 URL 或模型识别出的敏感文本。

### 8.2 内容安全

- 输入图片先于剧情生成进行安全检测；
- 输出案件进行第二次文本安全检测；
- 不生成针对照片中真实人物的犯罪、死亡或伤害叙事；
- 不生成自残、仇恨、色情、极端暴力或违法指导内容；
- MVP 定位为轻悬疑，不使用“真实预测”“灵异鉴定”等误导表述。

## 9. 数据与指标

### 9.1 核心漏斗

1. 进入首页；
2. 点击开始；
3. 选择或拍摄照片；
4. 照片通过预检；
5. 案件成功生成；
6. 发现第一条线索；
7. 收集全部线索；
8. 提交推理；
9. 查看真相；
10. 保存结果卡或再次拍摄。

### 9.2 MVP 成功指标

- 照片预检通过率 ≥ 70%；
- 案件成功生成率 ≥ 90%；
- 上传后 2 秒内出现可感知反馈的比例 ≥ 95%；
- 上传到可进入案件的中位时长 ≤ 15 秒；
- 案件完成率 ≥ 60%；
- 平均体验时长 2–5 分钟；
- 认为“线索确实与我的照片有关”的用户比例 ≥ 70%；
- 完成后再次拍摄率 ≥ 20%；
- 首轮收集至少 20 名真实用户与 10 条有效定性反馈。

### 9.3 护栏指标

- 安全检测漏放的已知违规样本为 0；
- 原图超期未删除事件为 0；
- 无法恢复且无解释的空白页面比例 < 1%；
- 单局 AI 定向修复次数不超过 1 次；
- 单局生成请求总时长上限为 30 秒。

### 9.4 关键事件

至少记录：

- `home_viewed`
- `start_clicked`
- `photo_selected`
- `photo_precheck_passed`
- `photo_precheck_failed`，附失败类型
- `generation_started`
- `generation_succeeded`
- `generation_failed`，附失败阶段
- `clue_opened`，附线索序号与耗时
- `all_clues_collected`
- `answer_submitted`，附尝试次数与正误
- `case_completed`
- `result_saved`
- `share_clicked`
- `replay_clicked`

事件不得包含照片、线索原文或其他可能识别用户的信息。

## 10. 非功能要求

### 10.1 性能

- 首页主要内容在普通移动网络下 2 秒内可交互；
- 图片压缩在客户端完成，避免上传原始超大图片；
- 案件生成中位时长目标 ≤ 15 秒，上限 30 秒；
- 游戏开始后不再调用模型，线索点击需即时响应；
- 结果卡在 2 秒内生成。

### 10.2 兼容性

- 优先适配手机竖屏；
- 支持近期版本的 iOS Safari、Android Chrome 和微信内置浏览器；
- 桌面端可体验，但不是首要验收对象；
- 浏览器不支持特定能力时，拍照降级为文件上传，分享降级为保存图片。

### 10.3 可恢复性

- 每局使用不可猜测的匿名会话 ID；
- 已生成的游戏状态保存在服务端；
- 客户端仅保存会话 ID 与当前展示状态；
- 生成任务、线索状态与答题次数均具备幂等处理；
- 页面刷新不应导致重复扣费或重复生成。

## 11. 页面清单

MVP 仅包含：

1. 首页；
2. 拍摄/上传页；
3. 图片预览与确认页；
4. 扫描等待页；
5. 图片不合格/生成失败页；
6. 案件介绍页；
7. 现场探索页；
8. 最终推理页；
9. 真相与结果卡页；
10. 隐私说明页。

## 12. 验收标准

### 12.1 主流程

- 用户无需注册即可从首页完成一局；
- 合格照片能生成包含三个可点击线索的案件；
- 三个线索均与照片中可辨识物体对应；
- 未收集全部线索时无法提交答案；
- 最终题只有一个程序定义的正确答案；
- 首次答错可重试一次；
- 所有用户最终都能看到真相页；
- 刷新页面后可恢复当前案件；
- 同一生成请求不会因重复点击而执行多次；
- 用户可保存结果卡并开始新案件。

### 12.2 异常流程

- 不合格照片获得具体、可执行的重拍提示；
- 违规照片不会进入生成流程；
- 生成超时后不无限等待；
- 坐标无效或答案不唯一的案件不会展示给用户；
- 定位置信度不足时可进入物品卡片降级模式；
- 模型、网络或结果卡失败时均有可恢复操作；
- 原始照片按约定时间删除，日志中不出现敏感图像数据。

## 13. 测试计划

### 13.1 图片测试集

准备至少 50 张获得授权的测试图片，覆盖：

- 桌面、宿舍、卧室、办公室、咖啡馆；
- 横图、竖图、不同手机尺寸；
- 正常、轻微模糊、过暗、过曝；
- 物体丰富、物体稀少、物体遮挡；
- 少量人物、多人主体、证件和屏幕文本；
- 截图、插画和违规测试样本。

### 13.2 自动化测试

- Schema 字段、类型、长度和枚举校验；
- 坐标范围与热点重叠校验；
- 唯一答案校验；
- 请求幂等与重复提交；
- 会话刷新恢复；
- 原图删除任务；
- 埋点字段不包含敏感内容。

### 13.3 人工体验测试

每个生成案件人工评分：

- 三个物体是否真实存在；
- 热点位置是否可接受；
- 每条线索是否与物体有关；
- 正确答案是否能由三条线索推出；
- 错误答案是否不会同样成立；
- 故事是否简短、自洽、适度悬疑；
- 是否愿意完成或再玩一次。

### 13.4 用户测试

- 第一轮：5 名用户可用性走查，修复阻断问题；
- 第二轮：20–50 名用户无指导体验；
- 记录漏斗数据，并在完成后询问三个问题：
  1. 哪个瞬间最有趣？
  2. 哪个地方让你困惑或出戏？
  3. 你愿意再拍一个场景吗？为什么？

## 14. 建议的实现模块

系统拆分为以下独立模块：

1. **图片接入层**：选择、拍摄、压缩、格式转换、上传；
2. **预检服务**：质量、安全、场景与物体数量判断；
3. **场景理解服务**：候选物体与归一化坐标；
4. **案件生成器**：将场景内容填入固定失踪事件结构；
5. **游戏校验器**：Schema、坐标、唯一答案与语义自洽校验；
6. **游戏状态服务**：会话、线索、答题次数、幂等与恢复；
7. **移动端游戏界面**：扫描、热点、线索卡、推理与结局；
8. **结果卡生成器**：固定模板绘制与保存；
9. **分析与运维**：匿名埋点、错误监控、成本和延迟统计；
10. **数据清理任务**：原始照片定时删除。

模块之间通过明确的结构化数据交互。替换模型供应商时，不应改变游戏状态与前端接口。

## 15. 迭代方向

只有当 MVP 数据证明用户认可“现实空间被转化为游戏”后，才考虑：

- 增加“异常物品”“顺序点击”等新题型；
- 支持奇幻、惊悚、治愈等主题；
- 允许用户选择难度与时长；
- 支持多张照片形成完整房间；
- 允许分享可复玩的案件链接；
- 支持朋友使用同一现场对战；
- 引入更精确的物体分割或 AR 交互。

迭代优先级由完成率、再次拍摄率、失败原因和用户访谈共同决定，不因技术可实现而自动加入。
